## Directory Purposes

**`include/`:**
- Purpose: All public C++ headers, organized by subsystem
- Contains: `.h` files only — classes, interfaces, structs, constants
- Key files: `AppConfig.h`, `FrameBuffer.h`, `MatrixLayout.h`, `BuildInfo.h`, `embedded_resources.h`

**`src/`:**
- Purpose: All C++ implementation files, mirroring `include/` structure
- Contains: `.cpp` files — one per header, organized in subsystem subdirectories
- Key files: `main.cpp` (entry point, global state, render loop)

**`frontend/`:**
- Purpose: Web UI — a Vite + TypeScript single-page application
- Contains: `.ts` source, `.html` entry, `.css` styles, dev mock server, Vite config
- Key files: `src/main.ts`, `mockApi.mjs`, `vite.config.ts`

**`test/`:**
- Purpose: PlatformIO native unit tests using Unity framework
- Contains: One directory per test suite, each with `test_*.cpp` files
- Key files: Each `test_*/` directory contains a standalone test binary

**`resources/dist/`:**
- Purpose: Frontend build output, consumed by `embed_resources.py` at firmware build time
- Contains: `index.html`, `script.js`, `styles.css` (Vite build artifacts)
- Generated: Yes (by `cd frontend && npm run build`)
- Committed: No (in `.gitignore`)

**`scripts/`:**
- Purpose: Pre-build Python scripts run by PlatformIO before compilation
- Contains: `generate_version.py` (git version stamping), `embed_resources.py` (asset embedding)

**`docs/`:**
- Purpose: Human-authored project documentation
- Contains: Architecture overview, DSL spec, LLM prompt template, release notes, roadmap, status

**`.planning/`:**
- Purpose: GSD workflow artifacts — phase plans, codebase maps, review documents
- Contains: `codebase/` (auto-generated analysis), phase directories

## Key File Locations

**Entry Points:**
- `src/main.cpp`: Firmware entry — `setup()` and `loop()` functions
- `frontend/src/main.ts`: Frontend entry — SPA initialization
- `frontend/index.html`: HTML shell served by ESP32 web server

**Configuration:**
- `platformio.ini`: Build environments, dependencies, build flags
- `include/AppConfig.h`: Hardware constants (panel size, pin assignments, timing intervals)
- `include/BuildInfo.h`: Version/channel/board identity (injected via build flags)
- `partitions_ota.csv`: ESP32 flash partition layout
- `frontend/vite.config.ts`: Frontend build configuration
- `frontend/tsconfig.json`: TypeScript compiler configuration

**Core Logic:**
- `src/main.cpp`: Render loop orchestration, global state, all service wiring
- `src/live/runtime/LiveProgramService.cpp`: DSL program lifecycle (validate, run, stop)
- `src/live/dsl/Parser.cpp`: Lux DSL parser
- `src/live/runtime/Executor.cpp`: Bytecode executor — evaluates expressions per pixel
- `src/MatrixLayout.cpp`: Logical→physical LED coordinate mapping
- `src/FrameBuffer.cpp`: Pixel buffer read/write

**Testing:**
- `test/`: 27 test suites, each in its own directory
- Test framework: PlatformIO `native` environment with Unity
- Run: `pio test -e native-test`

**Web API:**
- `src/web/LampWebServer.cpp`: HTTP route registration, request dispatch
- `src/web/LiveApi.cpp`: `/api/live/validate`, `/api/live/run`
- `src/web/PresetApi.cpp`: `/api/presets` CRUD
- `src/web/PlaylistApi.cpp`: `/api/playlists` CRUD
- `src/web/StatusJsonBuilder.cpp`: `/api/status` JSON aggregation
- `src/web/NetworkSettingsJson.cpp`: `/api/settings/network` GET/PUT
- `src/web/TimeSettingsJson.cpp`: `/api/settings/time` GET/PUT

## Naming Conventions

**Files:**
- PascalCase for class headers: `FrameBuffer.h`, `LiveProgramService.h`, `WiFiManager.h`
- Prefix `I` for interfaces: `IEffect.h`, `IWiFiAdapter.h`, `ISensorSource.h`, `ITimeSource.h`, `IFileStore.h`, `IFirmwareReleaseSource.h`, `IFirmwareInstaller.h`
- Prefix `Arduino` for ESP32-specific implementations: `ArduinoWiFiAdapter.h`, `ArduinoAht30SensorSource.h`, `ArduinoNtpTimeSource.h`, `ArduinoGitHubReleaseSource.h`
- Snake_case for scripts and config: `embed_resources.py`, `generate_version.py`, `platformio.ini`
- Test directories: `test_<module_name>/`

**Directories:**
- Lowercase, single-word: `effects/`, `live/`, `network/`, `sensors/`, `settings/`, `storage/`, `time/`, `update/`, `web/`
- Subdirectories mirror domain hierarchy: `live/dsl/`, `live/runtime/`

**Namespaces:**
- `lamp` — top-level project namespace
- `lamp::config` — hardware constants
- `lamp::effects` — built-in effect system
- `lamp::live` — live coding models and JSON
- `lamp::live::dsl` — DSL lexer, parser, AST
- `lamp::live::runtime` — DSL compiler, executor, services
- `lamp::network` — Wi-Fi management
- `lamp::sensors` — sensor abstraction
- `lamp::settings` — app settings and persistence
- `lamp::storage` — filesystem abstraction
- `lamp::time` — NTP time sync
- `lamp::update` — OTA firmware updates
- `lamp::web` — HTTP server and API handlers

**Functions:**
- camelCase: `toLinearIndex()`, `renderActive()`, `buildStatusSnapshot()`, `saveAndApplySettings()`
- Boolean queries prefixed with `is`/`has`: `isReady()`, `hasValidTime()`

**Variables:**
- Global state: `g_` prefix (`g_frameBuffer`, `g_effectRegistry`, `g_webServer`)
- Member fields: trailing underscore (`layout_`, `effects_`, `active_`, `fileStore_`)
- Constants: `k` prefix (`kPixelCount`, `kLogicalWidth`, `kLedDataPin`)

**Types:**
- Structs: PascalCase (`EffectContext`, `PlannedNetworkState`, `StatusSnapshot`, `RuntimeContext`)
- Enums: PascalCase with `k` prefix for values (`NetworkMode::kAccessPoint`, `ExpressionOp::kConstant`, `BlendMode::kNormal`)

## Where to Add New Code

**New Built-in Effect:**
- Header: `include/effects/YourEffect.h` — subclass `lamp::effects::IEffect`, implement `name()` and `render()`
- Implementation: `src/effects/YourEffect.cpp`
- Registration: Add global instance in `src/main.cpp`, call `g_effectRegistry.add(yourEffect)`
- Tests: `test/test_effects/`

**New DSL Feature (e.g., new expression function):**
- Token: `include/live/dsl/Token.h` — add token type if needed
- AST: `include/live/dsl/Ast.h` — add AST node if needed
- Lexer: `src/live/dsl/Lexer.cpp` — add tokenization rule
- Parser: `src/live/dsl/Parser.cpp` — add parse rule
- Compiler: `src/live/runtime/Compiler.cpp` — compile to bytecode op
- Executor: `src/live/runtime/Executor.cpp` — implement op evaluation
- Bytecode: `include/live/runtime/CompiledProgram.h` — add `ExpressionOp` enum value
- Tests: `test/test_dsl_parser/`, `test/test_dsl_executor/`

**New REST API Endpoint:**
- Handler: Add method to `LampWebServer` class in `include/web/LampWebServer.h` and implement in `src/web/LampWebServer.cpp`
- Or create a new `*Api.h`/`*Api.cpp` pair in `include/web/` and `src/web/`
- Register route in `LampWebServer::registerRoutes()`
- Tests: `test/test_*_api/`

**New Runtime Service (e.g., new sensor type):**
- Interface: `include/newdomain/INewSource.h`
- Implementation: `include/newdomain/ArduinoNewSource.h`, `src/newdomain/ArduinoNewSource.cpp`
- Service: `include/newdomain/NewRuntimeService.h`, `src/newdomain/NewRuntimeService.cpp`
- State struct: in service header
- Wiring: Add global instance in `src/main.cpp`, call from `loop()` at appropriate interval
- Tests: `test/test_new_service/`

**New Frontend Feature:**
- Primary code: `frontend/src/` — add TypeScript modules in existing subdirectories or create new ones
- Styles: `frontend/src/styles/app.css`
- Tests: `frontend/test/`

**Shared Utilities:**
- Config constants: `include/AppConfig.h`
- Build identity: injected via `-D` flags in `platformio.ini`, consumed via `include/BuildInfo.h`

## Special Directories

**`resources/dist/`:**
- Purpose: Frontend build output, auto-embedded into firmware
- Generated: Yes (by `npm run build` in `frontend/`)
- Committed: No

**`include/` (auto-generated files):**
- `embedded_resources.h`: Generated by `scripts/embed_resources.py` — gzipped PROGMEM arrays of frontend assets
- `BuildInfo.h`: Version macros injected via `-D` build flags from `platformio.ini`

**`.pio/`:**
- Purpose: PlatformIO build artifacts and library dependencies
- Generated: Yes
- Committed: No

**`.vite/`:**
- Purpose: Vite dev server cache
- Generated: Yes
- Committed: No
