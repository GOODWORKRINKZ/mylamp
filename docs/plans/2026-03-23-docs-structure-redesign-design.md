# Documentation Structure Redesign

## Goal

Сделать документацию проекта понятной с первого открытия: README должен быстро объяснять, что это за проект, как его запустить и куда идти дальше, а подробности по DSL и текущему статусу должны жить в отдельных документах.

## Considered Approaches

### 1. Один большой README

Плюсы:
- всё в одном месте;
- не нужно переключаться между файлами.

Минусы:
- README снова разрастается в длинную простыню;
- обзор, ограничения, DSL и roadmap смешиваются;
- сложнее поддерживать актуальность.

### 2. README как индекс + DSL и STATUS отдельно

Плюсы:
- быстрый вход для нового человека;
- DSL можно читать как отдельную спецификацию;
- статус проекта можно обновлять независимо от onboarding-части;
- лучше отражает реальное состояние продукта.

Минусы:
- появляется больше файлов;
- нужна аккуратная навигация между документами.

### 3. Разнести всё по множеству узких документов

Плюсы:
- очень чистое разделение тем;
- легко масштабировать при росте проекта.

Минусы:
- сейчас это избыточно;
- будет слишком много переходов для базовых задач.

## Recommendation

Берём подход 2. Он даёт нормальный баланс между обзорностью и деталями без лишней бюрократии в `docs/`.

## Target Structure

- `README.md` — входная точка, quick start, аппаратная схема, workflow, карта документации.
- `docs/DSL.md` — отдельная спецификация DSL: сущности, ограничения, примеры, рекомендации для LLM.
- `docs/STATUS.md` — честный срез текущего состояния: что готово, что частично готово, что ещё не реализовано.
- `docs/LLM_EFFECT_PROMPT.md` — готовый copy-paste prompt без изменений роли.

## Content Rules

### README

- короткий и визуально чистый;
- сначала ценность проекта, потом быстрый старт;
- подчёркивает реальные ограничения, но не превращается в backlog;
- активно ссылается на `docs/DSL.md`, `docs/STATUS.md` и `docs/LLM_EFFECT_PROMPT.md`.

### DSL Doc

- остаётся практической спецификацией, а не рекламным текстом;
- содержит минимальный шаблон и живой пример эффекта;
- фиксирует разрешённые конструкции и явные запреты.

### STATUS Doc

- отражает только подтверждённое состояние кода;
- отдельно отмечает mock/dev-возможности и реальные device-side ограничения;
- помогает не обещать лишнего.

## Verification

- проверить, что все новые ссылки в README ведут на существующие файлы;
- проверить, что README читается сверху вниз без повторов;
- проверить, что ключевое ограничение про `live/validate` и `live/run` на устройстве отражено в `docs/STATUS.md`.
